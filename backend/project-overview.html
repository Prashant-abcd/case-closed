Database Tables (PostgreSQL)

Hibernate auto-creates all tables on startup via ddl-auto: update. JSONB columns store raw JSON blobs — Hibernate 6 maps them via @JdbcTypeCode(SqlTypes.JSON).

users

  • id — UUID (PK)
  • username — VARCHAR
  • email — VARCHAR (unique)
  • password_hash — VARCHAR
  • created_at — TIMESTAMP
  • games_played — INT
  • games_won — INT

cases

  • id — UUID (PK)
  • user_id — UUID (FK → users)
  • status — ENUM (active/won/lost)
  • created_at — TIMESTAMP
  • completed_at — TIMESTAMP
  • truth_document — JSONB ← NEVER sent to frontend
  • case_briefing — JSONB
  • accusation_suspect_id — UUID
  • verdict — JSONB

suspects

  • id — UUID (PK)
  • case_id — UUID (FK → cases)
  • name, age, title, company
  • relationship_to_founder
  • avatar_seed — VARCHAR
  • interrogation_count — INT
  • is_killer — BOOLEAN ← NEVER sent to frontend

conversations

  • id — UUID (PK)
  • case_id — UUID (FK)
  • suspect_id — UUID (FK)
  • messages — JSONB array
  • started_at — TIMESTAMP
  • ended_at — TIMESTAMP

folders

  • id — UUID (PK)
  • case_id — UUID (FK)
  • suspect_id — UUID (FK)

  • intent_to_murder — TEXT

npc_interruptions

Step 1 — JPA Entities & Repositories

entity/User.java
@Entity

Maps to the users table. Stores player accounts. Passwords are never stored as plaintext — only the BCrypt hash.

@Entity
@Table(name = "users")
@Data @Builder @NoArgsConstructor @AllArgsConstructor
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(name = "password_hash", nullable = false)
    private String passwordHash;   // BCrypt hash, never raw password

    @Builder.Default
    private int gamesPlayed = 0;

    @Builder.Default
    private int gamesWon = 0;
}
entity/GameCase.java
@Entity

Maps to the cases table. Named GameCase because case is a reserved keyword in Java. The most important field is truthDocument — Claude's full solution to the case, stored as JSONB. It is never included in any API response.

@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "truth_document", columnDefinition = "jsonb")
private Map<String, Object> truthDocument;  // ← NEVER sent to frontend

@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "case_briefing", columnDefinition = "jsonb")
private Map<String, Object> caseBriefing;   // ← shown to player
entity/Suspect.java
@Entity

Each case has 5 suspects. The isKiller boolean is stored in the DB but never returned to the frontend. avatarSeed is used to generate deterministic DiceBear avatar images.

@Column(name = "is_killer", nullable = false)
private boolean isKiller;   // stored in DB, never returned to frontend

@Column(name = "avatar_seed")
private String avatarSeed;  // e.g. "suspect-vz-cto" — used for avatar URL
entity/Conversation.java  |  Folder.java  |  NpcInterruption.java
@Entity

Conversation — stores the full message history between player and a suspect as a JSONB array: [{role, content, timestamp}].

Folder — one per suspect per case. Player types free-form notes into contradictions and intent_to_murder text fields. These are submitted to the Lieutenant for verdict.

NpcInterruption — stores mid-interrogation phone call conversations with NPCs (Senator, lawyer, FBI agent). Triggered during interrogation, saved here.

repository/ (6 interfaces)
JpaRepository

Each entity has a Spring Data JPA repository. These interfaces give free CRUD methods with zero SQL. Custom query methods are derived from method names:

Step 2 — JWT Authentication

Every request (except /api/auth/**) must include a valid JWT in the Authorization: Bearer <token> header. Sessions are stateless — no cookies, no server-side session storage.

Auth Flow

  1. Register: Player sends username + email + password → password BCrypt-hashed → User saved → JWT generated → returned
  2. Login: Player sends email + password → AuthenticationManager validates → JWT generated → returned
  3. Every protected request: JwtAuthFilter reads header → extracts email from token → loads user → sets authentication in SecurityContext
service/JwtService.java
@Service

Handles all JWT operations. Uses the jjwt library. The secret key comes from application.yml. Tokens expire after 24 hours (86400000ms).

// Generate a token — subject is the user's email
public String generateToken(UserDetails userDetails) {
    return Jwts.builder()
        .subject(userDetails.getUsername())   // email
        .issuedAt(new Date())
        .expiration(new Date(System.currentTimeMillis() + expiration))
        .signWith(getSigningKey())
        .compact();
}

// Validate — checks email match AND expiry
public boolean isTokenValid(String token, UserDetails userDetails) {
    return extractEmail(token).equals(userDetails.getUsername())
        && !isTokenExpired(token);
}
filter/JwtAuthFilter.java
OncePerRequestFilter

Runs on every HTTP request exactly once. If a valid Bearer token is found, it loads the user and puts the authentication object into Spring's SecurityContextHolder — making the user "logged in" for the duration of that request.

String authHeader = request.getHeader("Authorization");
if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    filterChain.doFilter(request, response);  // no token → skip
    return;
}

String token = authHeader.substring(7);       // strip "Bearer "
String email = jwtService.extractEmail(token);

if (email != null && SecurityContextHolder.getContext().getAuthentication() == null) {
    UserDetails userDetails = userDetailsService.loadUserByUsername(email);
    if (jwtService.isTokenValid(token, userDetails)) {
        // Mark this request as authenticated
        SecurityContextHolder.getContext().setAuthentication(authToken);
    }
}
service/UserDetailsServiceImpl.java
UserDetailsService

Spring Security needs a way to load a user by username during authentication. We implement this interface and tell it to look up by email. This bridges Spring Security and our UserRepository.

@Override
public UserDetails loadUserByUsername(String email) {
    return userRepository.findByEmail(email)
        .map(user -> User.builder()
            .username(user.getEmail())       // Spring Security uses email as username
            .password(user.getPasswordHash())
            .build())
        .orElseThrow(() -> new UsernameNotFoundException("User not found"));
}
config/SecurityConfig.java
@Configuration

Configures Spring Security. Key decisions: CSRF disabled (stateless JWT doesn't need it), sessions are STATELESS, only /api/auth/** is public, everything else requires a valid JWT.

http
    .csrf(AbstractHttpConfigurer::disable)
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/auth/**").permitAll()  // register + login are open
        .anyRequest().authenticated()                 // everything else needs JWT
    )
    .sessionManagement(session -> session
        .sessionCreationPolicy(SessionCreationPolicy.STATELESS)  // no sessions
    )
    .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);

Step 3 — Case Generation (Claude AI)

When a player starts a new game, the backend calls Claude to generate a complete murder mystery from scratch. This happens once per game. The full truth is stored in the DB and never touched again by the frontend.

Case Creation Flow

  1. POST /api/cases — player hits the endpoint (JWT required)
  2. CaseService — gets current user from SecurityContext, calls CaseGenerationService
  3. CaseGenerationService — builds system + user prompts, calls ClaudeService
  4. ClaudeService — sends request to Anthropic API, returns raw JSON string
  5. CaseGenerationService — parses JSON into GeneratedCaseData object
  6. CaseService — saves GameCase + 5 Suspects + 5 empty Folders to DB
  7. CaseResponse — returned to frontend (no truthDocument, no isKiller)
service/ClaudeService.java
@Service

Thin wrapper around the Anthropic Java SDK. Reads the API key and model from config. The @PostConstruct method builds the client once on startup. Exposes a single generate(system, user) method.

@PostConstruct
public void init() {
    client = AnthropicOkHttpClient.builder()
        .apiKey(apiKey)   // from application-local.yml
        .build();
}

public String generate(String systemPrompt, String userPrompt) {
    MessageCreateParams params = MessageCreateParams.builder()
        .model(model)          // "claude-haiku-4-5" in dev
        .maxTokens(4096L)
        .system(systemPrompt)
        .addUserMessage(userPrompt)
        .build();

    Message response = client.messages().create(params);

    return response.content().stream()
        .flatMap(block -> block.text().stream())
        .map(textBlock -> textBlock.text())
        .reduce("", String::concat);
}
service/CaseGenerationService.java
@Service

Contains the prompt engineering. The system prompt tells Claude to act as a mystery writer and return only raw JSON. The user prompt defines the exact JSON schema Claude must follow — including 5 suspects, truth document, case briefing, and all the lies, contradictions and motives.

// System prompt — sets Claude's role
"You are a mystery writer generating a Silicon Valley murder mystery for a detective game.
 You must respond ONLY with valid JSON. No markdown, no explanation, no code blocks."

// User prompt — defines exact output schema
{
  "caseBriefing": { headline, articleBody, founderName, ... },
  "suspects": [{
    name, age, title, isKiller,
    trueAlibi,     // where they actually were
    lieTheyTell,   // what they tell the detective
    contradiction, // detail that exposes the lie
    motiveToKill
  }],
  "truthDocument": { killerName, weapon, method, crimeTimeline, ... }
}

// Parse response into typed Java object
return objectMapper.readValue(cleaned, GeneratedCaseData.class);
dto/generation/GeneratedCaseData.java & GeneratedSuspectData.java
Internal DTO

These are purely internal — Jackson deserializes Claude's JSON response into these POJOs. They are never serialized back to the frontend. @JsonIgnoreProperties(ignoreUnknown = true) means if Claude adds unexpected fields, the app won't crash.

service/CaseService.java
@Service

Orchestrates the full case creation. Runs in a single @Transactional block — if anything fails, nothing gets saved to the DB. Also creates empty Folder rows for each suspect upfront, so the player can immediately start taking notes.

@Transactional
public CaseResponse createCase() {
    User user = getCurrentUser();                    // from SecurityContext
    GeneratedCaseData generated = caseGenerationService.generateCase(); // calls Claude

    GameCase gameCase = GameCase.builder()
        .user(user)
        .truthDocument(generated.getTruthDocument()) // stored, never returned
        .caseBriefing(generated.getCaseBriefing())
        .build();
    caseRepository.save(gameCase);

    // Save 5 suspects + create empty folder for each
    suspects.forEach(suspect -> folderRepository.save(
        Folder.builder().gameCase(gameCase).suspect(suspect).build()
    ));

    user.setGamesPlayed(user.getGamesPlayed() + 1);
    userRepository.save(user);

    return CaseResponse.from(gameCase, suspectResponses); // safe — no secrets
}
